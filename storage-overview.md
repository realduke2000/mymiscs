## 存储概念

### 物理存储 HDD与SDD
||HDD|SDD|
|:---:|:---:|:---:|
|存储的基本原理|靠磁性物质存储数据，通过最小磁性单元磁畴磁化后的正反方向转化成电信号代表1/0，无磁状态为空|靠存储电子存储数据，隧道效应，通过加电压使得电子穿过绝缘体，困在浮栅中，无电子为1，有电子为0|
|数据组织方式|磁畴组成磁单元，多个磁单元组成扇区，扇区Sector是一个圆弧，最小的寻址单位，最早是512B, 现在是4KB。写数据的时候，优先在同一个柱面，也就是多个磁盘同一位置的扇区进行写数据，考虑到磁头大小和位置，柱面物理上可能不是绝对的垂直，不同的碟片上可能有偏移|由多个晶体管组成一个page，最小一般为16KB，多个page组成一个block，为最小擦除单位，然后组成die, plane|
|寻址方式|依靠磁头在磁盘的不同同心圆即磁道切换来寻址。|依靠互相垂直位线、字线加不同伏值的电压来寻址，最小单位是页Page，一般是16KB。由于和主流文件系统的最小单位4KB不同，ssd会复用page，可能将不同的逻辑块写入同一个page中，因此ssd维护一个逻辑块到物理地址的寻址映射表。(LBA - Logic Block Addressing, FBA - Fixed Block  Architecture)|
|物理接口|SATA, SAS|M.2(SATA, NVMe/PCIe)|
|读写语义|SCSI|SCSI/NVMe|

### IO基础

#### IOPS
每秒访问数，一般用4K, 8K小数据块模拟随机存储，HDD的IOPS为100 ~ 500, SDD的IOPS可以达到1K ~ 500K, 云存储采用多层内存缓存+sdd落地, IOPS可以达到更快。
目前数据中心存储采用以太网较多，传统企业使用的SAN网络较少，主要原因是价格高，FC兼容性差，无论是以太网还是FC，网络延迟对IOPS的影响都较小。
随机读写次数较多的，例如数据库，需要首先衡量存储设备的IOPS

#### 吞吐量(Throughput)
是指每秒数据的传输量，常用MB/s, GB/s单位。大文件，例如4K视频的读写，需要首先衡量存储设备的Throughput。

#### 延迟(Latency)
存储设备的相应速度，从收到请求到返回数据的时间，Latency对IOPS的影响较大，对小文件的随机读写，影响比较大。
HDD, 常用ms毫秒为单位, SSD，常用μs微秒为单位

#### 实际问题
Q: 假设你要为一个同时运行在线交易系统和视频点播服务的服务器选择存储，在线交易需要快速响应每笔订单，视频点播需要流畅播放高清视频，你会怎么平衡 IOPS、吞吐量和延迟？

A: 在线交易系统需求是低延迟高IOPS，在线视频网站的要求是低IOPS，高吞吐量。
在设计这个解决方案的时候，可以考虑将SSD主要用于交易系统，HDD主要用于视频点播服务，使用LVM+dm-cache实现。
将两个设备分别创建两个PV，加入VG中，创建交易系统卷的时候，指定物理卷为SSD,使用其中一部分块空间；创建视频点播服务卷的时候，将SSD物理卷中剩余的块空间作为cache pool，和HDD一起创建卷。
这样既满足了交易系统的高IOPS，也能满足视频点播服务的大容量存储，增加了SSD缓存cache，也提高了性能。

``` shell
pvcreate /dev/sdb  # SSD
pvcreate /dev/sdc  # HDD
vgcreate storage_vg /dev/sdb /dev/sdc

lvcreate -L 50G -n transaction_lv storage_vg /dev/sdb

lvcreate -L 50G -n ssd_cache_lv storage_vg /dev/sdb

lvcreate -l 100%FREE -n video_hdd_lv storage_vg /dev/sdc

lvconvert --type cache-pool --cachemode writethrough storage_vg/ssd_cache_lv

lvconvert --cache-pool storage_vg/ssd_cache_lv storage_vg/video_hdd_lv
```

### 存储不同部署方式
本地存储、网络存储、分布式存储

#### 本地存储
通过本机总线直连本地存储设备，延迟低、性能高，共享能力弱。
```
Application
        -> VFS
        -> File System(ext4, xfs)
        -> Block Layer
        -> Device Driver
        -> Disks

注: 有些应用可绕过文件系统直接使用块设备
Application(e.g. DB)
        -> Block Layer
        -> Device Driver
        -> Disks
```

#### 网络存储
通过网络(FC, Ethernet)连接存储设备，主机设备与存储设备分离，便于集中管理与共享。读写延迟、性能受到网络架构影响，传统网络存储通常以纵向扩展(scale up)为主。FC SAN需要专用网络，性能高，延迟低但价格昂贵。
##### 网络块存储
通过FC/iSCSI/NVMe-oF等方式，把远程的存储呈现为主机上的块设备。
主机通过NIC/HBA连接存储，需要一个主动的发现过程：触发后，主机侧的initiator发现远端的target/LUN, OS创建一个本地块存储设备(例如/dev/sdx)，应用对该设备的读写，通过initiator与远程的存储进行通信，完成真正的数据读写。这个连接存储设备的网卡，在操作系统中一直存在，也需要进行配置管理。
```
Application
        -> VFS
        -> File System
        -> Block Layer
        -> SAN/iSCSI/NVME-oF initiator
        -> NIC / HBA
    ~~~ Network ~~~
        -> Storage Array Controller
        -> Disks
```


#### 网络文件存储
通过以太网来连接的存储服务器，通过SMB/NFS协议提供文件服务，通过挂载将文件服务导出的目录挂载在本地目录进行读写。
```
Application
        -> VFS
        -> SMB/NFS Client
    ~~~~~~ Network ~~~~~~
        -> NAS Server
        -> Export/Share
        -> File System
        -> Disks

```

#### 分布式存储
通过软件方式，将多台机器上的存储资源组织成存储池，提供对象、块、文件等接口，具有横向扩展以及容错能力，系统复杂度较高。

##### Object对象
```
Application
        -> Network(S3/HTTP API)
        -> Object Storage Cluster
        +--> Storage Device on Node 1
        +--> Storage Device on Node 2
        +--> Storage Device on Node 3
```
##### File文件
```
Application
        -> VFS
        -> Distributed FS Client / SMB / NFS
    ~~~~~~ Network ~~~~~~
        -> Distributed File Storage Cluster
        +--> Storage Device on Node 1
        +--> Storage Device on Node 2
        +--> Storage Device on Node 3
```
##### Block块
```
Application
        -> VFS
        -> File System
        -> Block Layer
        -> Distributed Block Client / iSCSI / NVMe-oF Initiator
    ~~~~~~ Network ~~~~~~
        -> Distributed Block Storage Cluster / Network Protocol (iSCSI/NVMe-oF)
        +--> Storage Device on Node 1
        +--> Storage Device on Node 2
        +--> Storage Device on Node 3
```

### 数据保护技术
数据保护技术分为复制(副本)、快照、备份、归档四种。

#### 副本
Replica(副本)，把原始数据完整的保存多份，分散在不同的磁盘、结点甚至不同机房里，提高数据的可靠性和可用性。  
优势：  
* 提高可靠性，原始数据损坏，可以通过副本恢复。
* 提高可用性，原始数据损坏，可以通过副本来继续提供服务。
* 恢复速度快，原始数据损坏，使用副本完整复制就可以恢复，不需要计算校验等。
* 增加读性能，可以从不同副本同时读取数据，提高读性能。  

劣势：
* 空间效率低，空间利用率小于50%。
* 跨结点写增加网络开销。
* 多副本的一致性维护复杂度高。

适用场景：
* 高可用场景
* 小随机写写较多，延迟敏感
* 热数据、在线业务

副本常见的写入方式:
* 同步复制，一致性高，但延迟高
* 异步复制，延迟低，吞吐量大，但一致性维护复杂

副本的核心难点是什么？  
* 数据一致性问题  
  * 具体场景举例： 
    * 主副本写成功，次副本还没写完，主副本机器宕机
    * 有的副本收到了新数据，有的没收到
    * 客户端超时重试，可能造成重复写。  
  * 解决思路：
    * 强一致性使用同步写方式，副本写完后，主副本写入操作返回成功
    * 弱一致性在结点宕机恢复后，合并数据，解决数据版本分叉、冲突，也有的业务允许数据不一致，如加购物车、点赞数等。
    * 主副本复制时做标记，失败重传
    * 对写操作标序号/日志，已完成的写操作不再重复写。
  * 故障检测与切换问题  
    * 具体场景举例：
      * 副本是永久故障，还是由于网络抖动造成的临时故障。* 故障检测，分区后，两边都以为自己是“主”。
    * 解决思路：
      * 使用心跳，租期(lease)等检测成员状态。
      * 强一致性需求，写入前通过和副结点通信确认副结点可写入，确认主结点身份后再提交写入，返回成功。
      * 强一致性需求，使用分布式选举算法，raft, paxos等保证同一时刻只有单点可以处理写请求。
      * Fencing隔离，新主接管写请求后，把旧主踢出写路径。
  * 数据恢复与重建
    * 具体场景举例: 
      * 副本故障后补齐数据。
      * 网络抖动造成主副本复制数据失败。
    * 解决思路：
      * 全量同步，将数据同步到副结点。
      * 增量同步，使用DCM，位图记录数据脏区，只同步增量数据。
      * 快照+增量日志，先对其到某个快照，再重放快照后的写日志。
  * 性能与延迟
    * 具体场景举例：
      * 多副本系统，客户端可以从不同副本读取提高效率，那应该从哪个副本读最近、最新，负载最低
      * 写副本如何效率高
    * 解决思路：
      * 采用不同Loadbalance策略，如随机、轮询、最近等。
      * 读取数据中带版本或时间戳信息。
      * 中继策略，主副本只复制数据到中继节点，中继节点专门负责同步与其他副本的同步。
      * 链式复制，主副本写到副本1，副本1再写数据到副本2.
  * 扩容问题，数据可以从不同结点同步，同步可以发生在业务不忙的时段。
  * 副本放置策略，可以放在不同机架、机房甚至不同城市。

#### 具体数据保护技术
##### RAID
RAID全称Redundat Array of Independent Disks, 独立磁盘冗余阵列。是通过多块磁盘不同的组织形式，提高磁盘的读写效率以及可靠性的技术。
|名称|原理|特点|最少需要磁盘数|使用场景|
|:---:|:----:|:----:|:----:|:----:|
|RAID0|条带化，将不同数据分散写到不同磁盘上|提升读写效率，没有容错性，一块磁盘数据丢失就造成数据丢失|2|临时数据|
|RAID1|镜像，将数据同时写到不同磁盘上，提升数据可靠性|提升数据可靠性，有一块磁盘损坏数据依然可用；容量利用率低，50%|2|关键小容量数据，如系统盘|
|RAID2|将数据**按位**分散到不同磁盘，使用汉明码纠错机制，额外专用磁盘存储纠错码|实现复杂|3|实际场景几乎不使用|
|RAID3|将数据**按字节**分散到不同磁盘，额外专用磁盘存储纠错码|一块磁盘损坏数据依然可用，专用数据校验盘可能成为瓶颈|3|实际场景很少使用|
|RAID4|将数据**按块**分散到不同磁盘，额外专用磁盘存储纠错码|一块磁盘损坏数据依然可用，专用数据校验盘可能成为瓶颈|3|实际场景很少使用|
|RAID5|将数据**按块**分散到不同磁盘，校验信息也分散到不同磁盘上|一块磁盘损坏数据依然可用，读写性能、数据可靠性较均衡|3|通用文件存储|
|RAID6|将数据**按块**分散到不同磁盘，配备**两份独立校验信息**，校验信息也分散到不同磁盘上|读写性能较RAID5稍差，容错能力更强，两块磁盘损坏的情况下数据依然可用|4|通用文件存储|

##### 汉明码
Hamming Code(汉明码，海明码)，是一种基于奇偶校验思想构造的分组码，在原始数据中插入冗余校验位，使得接收方能够检测并纠正错位的。
标准汉明码(7, 4)校验位的计算方法：设原始数据长度为k，其校验位长度r需符合$2^r >= k + r + 1$ ,下表为最小校验码长度表。
|原数据长度k|最小校验位长度r|
|:---:|:----:|
|1 ~ 4|3|
|5 ~ 11|4|
|12 ~ 26|5|
|27 ~ 57|6|
|58 ~ 120|7|
|121 ~ 247|8|


组成: 汉明码H由校验位和原数据位共同组成，其中位序为1，2，4，8(即$2^i$)为校验位，其余位置为数据位。

构建: 校验位$Pn$位于$H$的$2^{i-1}$位，$Pn$的值与其他位序二进制表示中某一位为1的位的异或值为0。  
例如:  
P1，它与其他位序二进制表示最低位为1的第3(01**1**),5(10**1**),7(11**1**)...的异或值为0

校验: 收到数据后，对$Pn$进行校验，校验规则和构建规则相同，结果为$Sn$,校验正确$S_n$的值为0，错误为1，错误位$Wi$是$SnSn-1Sn-2...S1$构成二进制数，若值为0，表示数据正确，数据不需纠错；若不为0，则这个$Wi$位的数据错误，翻转该位的值。


|汉明码$H$十进制位|1|2|3|4|5|6|7|
|---|---|---|---|---|---|---|---|
|$H$二进制位|001|010|011|100|101|110|111|
|$H$位释义|$P_1$|$P_2$|$D_1$|$P_3$|$D_2$|$D_3$|$D_4$|
|原数据$D$|||1||0|1|1|
|校验位$P$算法|$H_{001} \oplus H_{011} \oplus H_{101} \oplus H_{111} = 0$|$H_{010} \oplus H_{011} \oplus H_{110} \oplus H_{111} = 0$||$H_{100} \oplus H_{101} \oplus H_{110} \oplus H_{111} = 0$||||
|$P$算法|$P_1 \oplus D_1 \oplus D_2 \oplus D_4 = 0$|$P_2 \oplus D_1 \oplus D_3 \oplus D_4 = 0$|1|$P_3 \oplus D_2 \oplus D_3 \oplus D_4 = 0$|0|1|1|
|$H$值|0|1|1|0|0|1|1|
|收到数据$H_{recv}$|0|1|1|1|0|1|1|
|校验过程|$S_1$=0|$S_2$=0||**$S_3$=1**|
|校正后数据$H_{recv}$|0|1|1|**0**|0|1|1|

备注:
总和校验值为$S_3S_2S_1$=$100$= $4$,因此第**4**位错误，翻转**1**为**0**，校验完成。

##### 纠删码
Erasure Coding(擦除码，纠删码)，通过计算并存储冗余数据获取容错能力的编码方法。  
纠删码常见的算法是RS(Reed-Solomon, 里德-所罗门)算法。RAID5/RAID6可以看做纠删码在存储中的应用。  
它的核心思想是把数据分成不同的k个块，再对这k个块的数据计算出m个校验块，一共得到k+m块数据。对于RS这种MDS码，只要这k+m块数据，还剩下任意的k块可用，就能完整的把数据恢复完整原始数据。  
其中：  
* `k`表示数据块数
* `m`表示校验块数
* 空间利用率为 $\frac{k}{k+m}$
* 最大可容忍同时丢失`m`个块
  
一般来说：
* `k`值越大，存储的空间利用率越高($\frac{k}{k+m}$)，但是k值过大导致编码小随机写，降级读(某些数据块、校验块不可用时提供的读服务，需要将数据重构后提供)，重建恢复的放大效应明显。
* `m`值越大，容错率越高，m值过大影响系统的性能、管理复杂度。
工程中常见的k,m的值: 中小规模 - 4+2， 6+2； 比较均衡：8+2， 8+3；超大规模：16+4。
在MinIO中，只需要设计EC（即m）的值，其Erasure set size(即分片总数N值， 2 ~ 16之间，默认为16)在初始化的时候由配置或系统自动根据拓扑确定，得出：k = N - m, 读quorum=k，写quorum=k+1.  
Ceph需显式指定k,m的值。  

**纠删码和副本实际问题**
1. 纠删码和副本有什么区别  
副本是原始数据的全量复制；纠删码是将原始数据的切片，并计算切片的校验码存储。  
副本空间效率较低，但数据恢复效率高；纠删码空间效率较高，但重建效率低。  
副本适合热数据；纠删码适用冷数据。
1. 为什么纠删码不适合小随机写
纠删码的小随机写场景，IO放大效果明显，每次写操作都需要重新读原数据切片并重新计算校验块再写入。

1. 为什么对象存储更爱用纠删码，而数据库不一定喜欢
对象存储写少读多，顺序写；数据库是在线热数据，小随机读写较多，不适合纠删码存储。

1. `k`和`m`应该怎么选？  
要根据空间利用率、容错要求、业务类型选择。  
工程常见4+2， 6+2, 8+2, 8+3等。m值越大，容错能力越强，但是管理、读写性能越弱；k值越大空间利用率越高，但小随机读、降级读性能越低。
1. RAID5/6和通用EC有什么区别?  
RAID5/6是EC在实际场景中的应用，RAID一般用于存储阵列，EC常用于分布式存储
1. 纠删码恢复时的代价是什么
数据重建时数据块和校验块的计算和重写过程中，磁盘的读放大；如果是分布式存储，网络开销会放大；CPU解码开销。
1. 纠删码能不能纠`错`，还是只能`纠失`?
不能，只能纠失。


##### VVR - Verias Volme Replicatior  

VVR是Veritas公司基于VxVM实现的数据复制子系统。  

首先需要了解VxVM的存储组织结构:  
Disks/LUN -> VxVM disks -> disk group -> subdisk -> plex -> volume  
1. 将系统中的磁盘，交由VxVM管理，成为VxVM disks
2. 将VxVM disks纳入一个disk group资源池
3. subdisk使得可以使用磁盘的一部分创建plex
4. plex通过不同的组织方式组成volume,提供给系统块服务,如concat,stripe,mirror,raid-5等。

**VVR概念**:  
`RVG`: Replication Volume Group, 由data volume, log volume和rlink组成，是一组需要保持写一致顺序的卷的集合。  
`SRL`: Storage replicator log， 用于存储写操作的环形缓存， 每个RVG中只能有一个SRL。  
`RLink`: Replication Link, 记录一个复制关系，包括链路状态，复制目标端的信息，复制模式等等，多个副本的情况中，每个副本对应一个rlink。  
`DCM`: Data change map，是一个bitmap，在SRL溢出后，用于记录data volume中的区域(region，默认为32k)数据变化，便面全盘同步。  

**复制过程**：  
在用户写入data volume的时候，VxVM会记录这个写操作，并先确保更新到SRL后，再写入data volume。SRL的内容会被复制到目标端的SRL中，再触发写操作，以此完成副本的复制。  
复制模式有同步和异步两种，  
同步模式： 在本地写入SRL后，会通知复制目标端完成SRL的更新，再返回写成功。  
异步模式： 在本地写入SRL后，即返回写成功，再后台完成SRL的同步。

**SRL缓存溢出问题**:  
当SRL溢出时，会启用DCM来标记哪些区域发生过变化，在链路回复后进行增量同步，避免全量重传的高昂代价。
